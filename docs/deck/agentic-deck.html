Phase 01 — Foundation · Anatomy

Five components. Each with one clear responsibility.

Skill — The Hands

Procedural instructions executed in the caller's context. The Type O skill is the only supported entry point — it owns routing, context relay, and the convergence loop.

Two types
O · Orchestrator — user entry; owns the session or does its own work
P · Procedure — agent-called; thin, non-interactive, no branching
A persona without a Type O skill is incomplete.
"Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to its toolkit." code.claude.com/docs/en/skills
Agent — The Brain

Reasoning specialist in its own isolated context window. Assesses, decides, and returns structured output. Never spawns agents directly — that belongs to the calling skill.

Anatomy
Input — declared params; fail fast on missing
Knowledge — Grep-loaded on demand; specialization by loading
Reasoning — LLM thinking, deciding, branching
Output — structured; Glob+Grep verified
Modes — one body, multiple lean invocation contexts
Platform-agnostic. Workers call skills — skills know the platform.
"Subagents are specialized AI assistants that handle specific types of tasks. Each runs in its own context window with a custom system prompt, specific tool access, and independent permissions." code.claude.com/docs/en/sub-agents
MCP — The Reach

Structured tool calls that reach external systems — no file reads needed. Jira tickets, IDE diagnostics, Figma designs, and build results returned directly to the agent.

Examples
Atlassian — Jira · Confluence · Bitbucket
ide — diagnostics · code execution
Figma · XcodeBuildMCP — design + build
Skills reach the codebase. MCP reaches everything else.
"MCP servers give Claude Code access to your tools, databases, and APIs. Once connected, Claude can read and act on that system directly instead of working from what you paste." code.claude.com/docs/en/mcp
Reference — The Knowledge

Deep knowledge loaded on demand via Grep — never embedded in agent bodies. Update one reference file, every agent that reads it gets the update automatically.

Three tiers
CLAUDE.md — universal rules, always loaded
Agent body — decision logic for that agent only
Reference docs — deep knowledge, one Grep per section
Core = what. Platform = how in that language.
"CLAUDE.md files give Claude persistent instructions for a project. Keep it to facts Claude should hold in every session." No official doc for deep reference docs — this pattern extends the memory system. code.claude.com/docs/en/memory
Hooks — The Automation

Shell commands that fire on lifecycle events with no model involvement. Logic that must always run — regardless of what the agent decides to do.

Four events
PreToolUse — intercept before a tool runs
PostToolUse — react after a tool completes
Stop — run when the agent finishes
Notification — fire on agent events
For validation and safety guards — not logic that belongs in a skill.
"Hooks let you execute shell commands at key points in Claude's lifecycle — before or after tools run, when the agent stops, or on notifications." code.claude.com/docs/en/hooks

The Orchestrator skill is the workflow. Agents are just steps it coordinates.

It owns
The session

Routing — resume vs new. Context pre-loading. Approval gates. The loop that drives agents forward. The engineer interacts with the skill, not the agents.

It delegates

Decisions go to the Strategist. Code generation goes to the Worker. Each agent runs in its own isolated context — the skill only sees the structured result.

Its constraint
The main context

The skill runs in your session — the same context window you're in. Every round of agent results lands here. Design to keep rounds low.

Phase 01 — Foundation · Orchestrator Skill

The Orchestrator skill runs in your context window. Three things it can do — and one constraint that shapes all of them.

Capability 1 — Parallel agents

One skill step can spawn N agents simultaneously. Each runs in its own isolated context and returns findings. Same wall-clock time as spawning one.

# spawn domain + data + pres planners
# in the same skill step
Agent(domain-planner, background: true)
Agent(data-planner,   background: true)
Agent(pres-planner,   background: true)
← all three run in parallel
Capability 2 — Convergence loop

The skill owns the loop. It spawns agents, collects decisions, and loops back until the strategist signals convergence. Each agent call is self-contained.

while not converged:
  result = Agent(strategist, findings)
  # Decision: spawn-planners
  findings = spawn_planners_parallel()
  # Decision: converged
  → break
Constraint — Main context window

Every agent result returns to the skill — which lives in your main session. Each round adds more history. When context fills, Claude compacts it into a summary.

Compaction = reduced quality. The summary loses detail. Subsequent rounds reason on a lossy view of earlier decisions.
Design for minimal rounds. Write findings to disk — pass paths, not content. Every byte returned to the skill is a byte closer to compaction.
Agents run in isolated contexts — they never see the main session. But the skill does, and every call back to it has a cost. Parallel spawning and disk-based findings are how we keep the round count low.

Phase 01 — Foundation · Agents

Five parts. Every agent is built from the same anatomy.

01
Input

What the agent receives to start — mode, feature name, platform, file paths. Declared explicitly. Missing input = immediate stop.

02
Knowledge

Reference docs and patterns loaded on demand. Each agent greps only what its task needs — nothing more. Specialization is a loading decision.

03
Reasoning

The LLM applies thinking, deciding, and branching to everything it has loaded. This is the part no fixed script can replace — it handles ambiguity and edge cases.

04
Output

Declared and structured — a Decision: block, a ## Findings section, verified file paths. The caller routes on it without ambiguity.

05
Modes

One agent, many invocation contexts. Each mode loads only its own instruction block. The agent never reads instructions for modes it isn't in.

Isolated context window keeps all five parts — and everything they touch — out of your main session. The main session only ever sees the structured output.

Phase 01 — Foundation · Reference

One source of truth. Every agent reads it. Nobody copies it.

Without reference docs
  • Architecture patterns embedded inside every agent
  • One convention change → update 40 files
  • Agents drift from each other over time
  • New platform means rewriting agent bodies
With reference docs
  • Patterns live in one file per topic per platform
  • Agents grep exactly the section they need — nothing more
  • Update once → every agent that reads it gets the update
  • New platform adds a new impl file — no agent changes
Reference docs are the team's shared knowledge base — the architectural decisions your team has made, written once, applied consistently by every agent on every session. It's what makes the system teachable and maintainable at scale.
Phase 01 — Foundation · Reference

Topic. Term. Three reference types. One grep key.

Process Implementation · Design · Requirement · … │ Topic domain · data · presentation · design-system · … │ one reference file per topic per platform │ Term UseCase · Entity · Repository · MpButton · … │ one ## heading per term, same name everywhere │ ├── -theory.md what it IS · platform-agnostic ├── -impl.md how to build it in this language └── -catalog.md queryable symbol inventory (### per entry)
The Term is the grep key
One Term = one canonical name = one ## heading, everywhere.

Catalog entries use ### per symbol — agents grep the symbol name directly, read 8 lines. No full-file load, ever.

Phase 04 — Expansion · Design Evolution

We didn't design it all at once. Each problem taught us the next piece.

Origin
Raw Skill

SKILL.md in the main context. Engineer types, skill runs. No agents, no isolation. Worked fine for simple, bounded tasks.

Problem 1
Context fills

As the skill grows, the main context fills. Quality degrades. The skill can't reason well on a compacted history.

Solution

Split to Agents — isolated context windows. Skill orchestrates; agent reasons. Only the result returns to main context.

Problem 2
Agents bloat

Agents accumulate every pattern and convention — unscoped. They hit their own context limit. Too much knowledge, not enough relevance.

Solution

Reference docs + Grep. Agents load only the sections their task needs. Multiple scoped agents — each expert in one layer.

Problem 3
One pass isn't enough

Domain, data, and presentation layers depend on each other. A single agent pass misses cross-layer impact. Output is incomplete.

Solution

Convergence loop. Planners report cross-layer impact. Strategist decides: more rounds or converged? Loop until stable.

Problem 4
Repeating ourselves

Agents writing the same create-entity boilerplate inline, per platform. No reuse. Update the pattern → update every agent.

Solution

Type P skills called by agents — not the Orchestrator. Platform-specific. One definition, updated once.

Phase 04 — Expansion · Building an Orchestrator

Four questions. Answer them before writing a single instruction.

01
What's the goal?
→ Output

Define the structured result before writing any instructions. What does the skill expect back from agents? Decision blocks, findings format, verified file paths. Declare it first — the routing logic depends on it.

02
What does it need?
→ Input

What must the engineer provide? Mode, feature name, platform, file paths. Declare every required parameter explicitly. Missing input = MISSING INPUT: <param> immediately — no guessing, no defaults.

03
How does it run? Who's involved?
→ Agents + Loop

Do you need a convergence loop? Which agents reason about what? Define each agent's modes, knowledge scope, and how they communicate — Decision blocks in, findings files on disk out. Never pass content inline between rounds.

04
Will 200K hold?
→ Context budget

Pro context is 200K tokens. Estimate: rounds × agent output size. If the workflow needs many rounds or returns large results, split into a second Orchestrator. One skill's structured output becomes the next one's input.

The order matters: Output → Input → Process → Budget. Designing output first prevents the skill from becoming a black box that returns whatever the agent feels like.

Phase 04 — Expansion · Limitations

Why not end-to-end? Four reasons human review still matters.

01
Context window

A full end-to-end workflow — across multiple roles and hand-offs — fills the main context. Once compacted, the session reasons on a lossy summary. Quality at the final step is lower than at the first.

02
Token billing

A billing cap or timeout mid-workflow orphans the Orchestrator skill. There is no clean auto-recovery — a human has to re-enter, re-read state, and decide how to continue. The longer the chain, the higher the risk.

03
Supervision

Agents aren't mature enough to run unsupervised across team boundaries. Each hand-off is a decision point — does the RFC accurately reflect the Epic? Does the plan match the RFC? A human eye at each gate catches drift early.

04
Gaps

There is no agreed input/output contract between Product → Design → Developer → QA yet. We're still mapping what each team produces and what the next team needs. Agents can't bridge a gap that hasn't been defined.

Each limitation is a research problem, not a dead end. Context efficiency, resume protocols, maturity metrics, and cross-team I/O contracts are all active work.
