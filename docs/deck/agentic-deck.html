Context

Software engineering was never about coding. AI just made it obvious.

Before AI
With AI
Coding was the slowest, largest, and most expensive part of building software.
AI writes code. Coding is no longer the bottleneck.
Engineers spent most of their time typing — so software engineering looked like typing.
The bottleneck shifted to the idea: what system to build, how it scales, how it holds under load.
"Software engineering = coding." That assumption made sense then. Coding was the constraint.
Software engineering = building stable, scalable systems aligned with product needs. It always was.
AI didn't change what software engineering is. It confirmed what it always was. The engineer's job is now fully about the system — the idea, the architecture, the decisions. The code follows.
The Shift

Every discipline had its own "coding". AI just removed all of them.

Phase 01 — Foundation · Anatomy

Five components. Each with one clear responsibility.

Skill — The Hands

Procedural instructions executed in the caller's context. The Type W skill is the only supported entry point — it owns routing, context relay, and the convergence loop.

Two types
W · Workflow — user entry; owns the session or does its own work
P · Procedure — agent-called; thin, non-interactive, no branching
A persona without a W skill is incomplete.
"Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to its toolkit." code.claude.com/docs/en/skills
Agent — The Brain

Reasoning specialist in its own isolated context window. Assesses, decides, and returns structured output. Never spawns agents directly — that belongs to the calling skill.

Three roles
Orchestrator — returns Decision blocks; never writes files
Planner — read-only layer explorer; returns findings
Worker — executes the plan; writes source files via skills
Platform-agnostic. Workers call skills — skills know the platform.
"Subagents are specialized AI assistants that handle specific types of tasks. Each runs in its own context window with a custom system prompt, specific tool access, and independent permissions." code.claude.com/docs/en/sub-agents
MCP — The Reach

Structured tool calls that reach external systems — no file reads needed. Jira tickets, IDE diagnostics, Figma designs, and build results returned directly to the agent.

Examples
Atlassian — Jira · Confluence · Bitbucket
ide — diagnostics · code execution
Figma · XcodeBuildMCP — design + build
Skills reach the codebase. MCP reaches everything else.
"MCP servers give Claude Code access to your tools, databases, and APIs. Once connected, Claude can read and act on that system directly instead of working from what you paste." code.claude.com/docs/en/mcp
Reference — The Knowledge

Deep knowledge loaded on demand via Grep — never embedded in agent bodies. Update one reference file, every agent that reads it gets the update automatically.

Three tiers
CLAUDE.md — universal rules, always loaded
Agent body — decision logic for that agent only
Reference docs — deep knowledge, one Grep per section
Core = what. Platform = how in that language.
"CLAUDE.md files give Claude persistent instructions for a project. Keep it to facts Claude should hold in every session." No official doc for deep reference docs — this pattern extends the memory system. code.claude.com/docs/en/memory
Hooks — The Automation

Shell commands that fire on lifecycle events with no model involvement. Logic that must always run — regardless of what the agent decides to do.

Four events
PreToolUse — intercept before a tool runs
PostToolUse — react after a tool completes
Stop — run when the agent finishes
Notification — fire on agent events
For validation and safety guards — not logic that belongs in a skill.
"Hooks let you execute shell commands at key points in Claude's lifecycle — before or after tools run, when the agent stops, or on notifications." code.claude.com/docs/en/hooks

Phase 01 — Foundation · Runtime Flow

How the components connect at runtime.

User
describes what to build
Skill (W) — entry

Orchestrator (Agent) — isolated context
reasons about intent · returns Decision: spawn-planners · never spawns agents or writes files

spawns Planners in parallel per round · accumulates findings · feeds back to Orchestrator · loops until Decision: converged

code  ·  MCP → external systems

Phase 01 — Foundation · Skills

Two types. Each with a different job, a different caller.

W · Workflow — user entry, owns the session
--- builder-plan-feature ---
user-invocable: true
allowed-tools: Agent, AskUserQuestion, Read

# Step 1 — Gather Intent
Spawn orchestrator (gather-intent)
→ returns Decision: spawn-planners

# Step 2 — Convergence Loop
Spawn planners in parallel per round
Send findings → orchestrator
Loop until converged or blocked

# Step 3 — Synthesize
Spawn orchestrator (synthesize)
→ writes plan.md + context.md

# Step 4 — Approve
AskUserQuestion
  Approve  → execute
  Discuss  → revise → re-approve
  Discard  → delete run, stop

# Step 5 — Execute
Spawn feature-worker
  plan.md + context.md inlined

# Simple W skills do their own work
(no Agent tool needed — Read, Edit, Ask)
P · Procedure — agent calls, thin and non-interactive
--- builder-domain-create-entity ---
user-invocable: false
allowed-tools: Read, Grep, Bash
(no AskUserQuestion — no branching logic)

# Input
entity_name, target_path, module

# Step 1 — Load reference (Grep-first)
Grep "^## Entity" in domain.md
Read offset + limit ← section only
Never the full file

# Step 2 — Check preconditions
Glob target path
STOP if file already exists

# Step 3 — Write
Create entity from reference pattern
Logic belongs in the worker — not here

# Output
Glob + Grep verified path only
Start with Type W for any user-facing entry — simple ones do their own work, complex ones delegate to agents. Type P when an agent needs a non-user-invocable, thin step with no branching or runtime complexity — logic belongs in the worker, not the skill.
Phase 01 — Foundation · Agents

An agent is only as good as what it loads and how it reasons.

Orchestrator — Tech Lead
# Personality
Reasons about the big picture.
Decides what runs, not how.

# Load first
Intent + accumulated findings
(passed inline by trigger skill)

# How to think
What does this feature need?
Which layers are affected?
Is there enough info to act?

# Decision output
Decision: spawn-planners
  spawn: [domain, data]
  reason: …

Decision: converged
Decision: blocked
  question: …

Never: spawn agents, write files
Planner — Layer Expert
# Personality
Specialist in one CLEAN layer.
Read-only. Reports. Never acts.

# Load first
Grep scope-matched reference section
(domain, data, pres, or app only)

# How to think
What already exists in this layer?
What needs to change or be created?
What other layers will be affected?

# Decision output
## Findings
  - existing artifacts
  - gaps and changes needed

### Impact Recommendations
  data: [required] ← affects next round
  app:  [required]

Never: write files, leave its layer
Worker — Craftsman
# Personality
Executes precisely. Verifies output.
Calls skills — never guesses platform.

# Load first
plan.md + context.md
(inlined by trigger skill — no cold reads)

# How to think
Does this artifact already exist?
Which skill creates this type?
What does reference say for this layer?

# Load reference on demand
Grep "^## Entity" in domain.md
Read offset=42 limit=15  ← section only

# Output
## Output
  Glob + Grep verified paths only

Never: re-read, skip skill, guess layer
Two kinds of knowledge to load
lib/core/reference/         ← load first
  What is a UseCase?
  What belongs in Domain?
  Generic. Definition. Principle.

lib/platforms/<p>/reference/ ← load second
  How to write a UseCase in Swift?
  Which pattern does Flutter use?
  Specific. Implementation. Syntax.
knowledge-query — never load the full file
# section-query  (reference doc section)
Grep: "^## Entity" in domain-impl.md → line 42
Read offset=42 limit=15   ← N from  annotation
→ 15 lines. Section only. Not 200.

# symbol-query  (class / function body)
Grep: "class GetUserUseCase" → line 18
Read offset=13 limit=60   ← window around symbol
→ body only. Expand only if class exceeds window.

# Read-once rule
Never re-read a file seen this session
If an agent isn't performing — check what it loaded. Poor output almost always means missing or wrong context, not a reasoning failure. Fix the knowledge, fix the agent.
Phase 01 — Foundation · Reference

Topic. Term. The two levels every reference doc must have.

Process Implementation · Design · Requirement · … │ Topic domain · data · presentation · components · … │ one reference file per topic per platform │ Term UseCase · Entity · Repository · Button · … │ one ## heading per term, same name everywhere │ ├── Core definition what it IS · platform-agnostic └── Platform impl how to build it in this language
The Term is the grep key
One Term = one canonical name = one ## heading, everywhere.

If iOS uses ## Use Case Protocol instead of ## UseCase, the grep misses — agent gets incomplete knowledge. Platform dialect belongs in the body, never the heading.
Core definition vs Platform implementation
lib/core/reference/code-architecture/domain-theory.md
────────────────── answers: WHAT and WHY ──────────────────
## UseCase 
A use case encapsulates one business operation.
It depends on nothing outside the domain layer.
Invariants: no framework imports, single execute()...
(28 lines — agent reads exactly this section)

## Entity 
An entity is a pure domain object with identity.
No serialization logic. No platform dependencies...


lib/platforms/ios/reference/code-architecture/domain-impl.md
────────────────── answers: HOW in Swift ──────────────────
## UseCase   ← same Term, Swift body
final class GetUserUseCase: UseCase {
  struct Params { let userId: String }
  func execute(_ params: Params) async throws
  → Swift protocol, Needle DI, async/await

## Entity   ← same Term, Swift body
struct UserEntity: Equatable { ... }
Core definition answers what and why — platform-agnostic, shared by every platform. Platform implementation answers how in that language. A worker Greps the core file to understand the contract, then Greps the platform file to write the code. One Grep each. Never the full file.
Phase 01 — Foundation · MCP

Skills reach the codebase. MCP reaches everything else.

Two directions of reach
Via Skill — into the codebase
# Worker calls a platform-contract skill
call builder-domain-create-entity
  feature: GetUser
  platform: ios

→ creates GetUserUseCase.swift
Via MCP — into external systems
# Agent calls an MCP tool directly
mcp__atlassian__get_jira_issue(
  issue_key: "PROJ-42"
)

→ returns ticket, context, acceptance criteria
MCP servers in use today
ServerReaches into
Atlassian Jira tickets · Confluence pages · Bitbucket PRs
ide Live diagnostics · in-editor code execution
Figma Design data · component specs · asset export
XcodeBuildMCP Build · test · simulate · screenshot on iOS
MCP tools are just tool calls — no special wiring in the agent. Any agent with the right MCP server configured can reach any external system.
